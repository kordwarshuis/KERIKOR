# Public / private key pair in centralised versus decentralised context

Explain the difference between the public / private key pair in a centralised context (https) and a decentralized context (Bitcoin, and a step further, KERI)

## Answer (KERI wizard)

At a high level, all three systems use **public/private key cryptography**, but they use it for very different purposes.

The key distinction is:

* In **centralized systems** like HTTPS, keys establish trust through a trusted authority.
* In **decentralized systems** like Bitcoin, keys establish ownership and authorization without a central authority.
* In **KERI**, keys become part of a self-sovereign identity system where trust is rooted in cryptographic events and key continuity over time.

---

## 1. Centralized Context: HTTPS / TLS

Relevant concepts:

* Public Key Cryptography
* Transport Layer Security
* Certificate Authority

In HTTPS, the public/private key pair is mainly used to:

1. **Authenticate a server**
2. **Secure communication**

### How it works

Suppose you visit a bank website.

The server has:

* a **private key** (secret)
* a **public key** (shared via a certificate)

When your browser connects:

* the server proves it owns the private key
* your browser verifies this using the public key
* a secure encrypted session is established

But here's the critical part:

### Trust comes from a central authority

Your browser does **not inherently trust the server’s public key**.

Instead, it trusts:

* a Certificate Authority like DigiCert or Let’s Encrypt
* which signs the server certificate

So the trust chain is:

```text
Browser trusts CA
CA trusts website
Therefore browser trusts website
```

This is fundamentally centralized.

### Role of the keys

#### Private key

* proves server identity
* decrypts/authenticates TLS handshake data
* must remain secret

#### Public key

* distributed openly
* embedded in certificates
* used by clients to verify signatures

### Important property

The keypair is tied to:

* a domain name
* an organization
* a certificate hierarchy

Identity is externally assigned.

---

## 2. Decentralized Context: Bitcoin

Relevant concepts:

* Bitcoin
* Digital Signature
* Blockchain

Bitcoin uses the same mathematical primitives, but the purpose changes dramatically.

The keypair is no longer about:

* securing a communication channel

Instead it is about:

* controlling assets
* authorizing state changes

---

### In Bitcoin

Your private key:

* gives you spending authority

Your public key:

* lets others verify your authorization

No central authority is needed.

---

### Example

If you own Bitcoin:

```text
Private key → signs transaction
Network → verifies signature using public key
```

If valid:

* the network accepts the transaction
* ownership transfers

No bank needed.

---

## Core Difference from HTTPS

### HTTPS

Keys answer:

> “Can I trust this server?”

### Bitcoin

Keys answer:

> “Is this transaction authorized by the asset owner?”

---

## Identity model

In Bitcoin:

* identity is *pseudonymous*
* keys ARE the identity

There is no:

* certificate authority
* central registry
* username/password system

Possession of the private key equals control.

---

## Trust model

### HTTPS

Trust is hierarchical:

```text
Root CA → Intermediate CA → Website
```

### Bitcoin

Trust is distributed:

```text
Consensus + cryptographic verification
```

Nodes independently verify signatures.

No trusted intermediary exists.

---

## 3. KERI: Key Event Receipt Infrastructure

Relevant concepts:

* Trust over IP Foundation
* Self-Sovereign Identity

KERI takes decentralized key management much further.

Bitcoin proved:

* keys can control money without central authorities

KERI asks:

> Can keys control identity itself without central authorities?

---

## The problem KERI tries to solve

Bitcoin identities are fragile:

If a private key is:

* lost → identity/assets lost
* stolen → attacker becomes you

There is no native concept of:

* key rotation
* recoverable identity
* persistent decentralized identifiers

KERI introduces:

* **key state continuity**
* **rotatable identifiers**
* **cryptographically verifiable identity histories**

---

## KERI’s core idea

Identity is not a static public key.

Identity is:

> a verifiable sequence of key events over time

Examples of events:

* inception
* rotation
* delegation
* recovery
* revocation

---

## Comparison

### HTTPS

```text
Identity = certificate signed by CA
```

### Bitcoin

```text
Identity = possession of private key
```

### KERI

```text
Identity = continuous cryptographic event history
```

---

## Why this matters

In Bitcoin:

* rotating keys is awkward
* identity continuity is weak

In HTTPS:

* identity depends on authorities

KERI tries to achieve:

```text
Decentralized identity
+
Key rotation
+
Recovery
+
Auditability
+
No central root authority
```

---

## Deeper conceptual shift

### HTTPS: “Trusted Third Party”

Trust anchor:

* certificate authority

The public key is trusted because:

* someone authoritative signed it

---

### Bitcoin: “Trust the Ledger”

Trust anchor:

* decentralized consensus

The public key is trusted because:

* signatures validate against ledger rules

---

### KERI: “Trust the Key Event Log”

Trust anchor:

* cryptographic continuity

The current key is trusted because:

* there is a verifiable chain of prior key rotations and receipts

This is closer to:

* a decentralized passport system
  than to:
* a payment network

---

## Mental Model

### HTTPS

```text
Who are you?
→ CA says I’m legitimate.
```

---

### Bitcoin

```text
Who are you?
→ Whoever controls this private key.
```

---

### KERI

```text
Who are you?
→ The controller of this continuously verifiable identifier history.
```

---

## One More Important Distinction

In HTTPS:

* keys are usually server infrastructure

Users rarely interact directly with them.

In Bitcoin/KERI:

* keys belong to users/agents directly
* key custody becomes central

This changes:

* UX
* security
* recovery
* governance
* responsibility

dramatically.
